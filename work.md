## 响应时机选择（何时响应）

### **延时策略：**

当前决策：采用**固定阈值**（如 200-500ms），优先保障稳定性与确定性

未来探索：后续可评估基于置信度的”动态阈值”策略（即高置信度瞬时响应，低置信度延迟等待以容错），但在当前阶段暂不引入，避免增加不确定性

### 打断机制（Barge-in）

**当前决策**：**暂不启用主动打断（打断用户）**

尽管打断能提升沟通效率，但在小模型算力受限且多路语音环境复杂的现状下，误判风险极高（错误打断会造成极差的用户体验）。当前策略严格遵循”话轮结束”原则，即等待用户语音完全结束后再触发响应

4、对简单01概率的改进：Logits 提取法

LLM 的最后一层会输出全量词表的得分（Logits）。通过对”1”和”0”这两个特定 Token 的 Logits 进行 Softmax，可以得到一个相对概率。设 $L_1$ 为 Token “1” 的 Logit，$L_0$ 为 Token “0” 的 Logit，则响应概率 $P$ 为：

$$
P(respond) = \frac{e^{L_1}}{e^{L_1} + e^{L_0}}
$$

### 1 对话长度建模与分布策略

- **对话会话（Session）的定义**

在多人车载场景中，连续的语音流通过**沉默阈值（Silence Threshold,** $T_{gap}$**）**进行切分。设定 $T_{gap} = 10s$，若相邻两句话的时间间隔 $\Delta t > T_{gap}$，则判定为两个独立的会话（Session）。

定义”多人对话”为车内全员至少参与一次发言。

- **长度衡量指标**

传统的”轮次（Turn）“在多人抢话、叠音场景下难以界定。因此，本方案在单一 Session 内，采用**话语总条数**作为该段对话复杂度的衡量指标。

- **泊松分布建模依据**

经过沉默阈值切分后，车载场景下的独立会话具有”阵发、短促”的特点。对于含 $N$ 人的参与群体，设定单场 Session 的话语条数服从参数为 $\lambda = 3 \times N$ 的泊松分布。该分布精准模拟了真实场景中”单次交互以中短长度为主，且随条数增加概率呈阶乘级迅速衰减”的物理特性。

```
4. **关键：时间戳生成逻辑**：
   - 语速：3-4字/秒。
   - 基础停顿：话轮间隙 0.3-1.5s。
   - **抢话/打断**：若逻辑紧凑，下一句的 start 可以比上一句的 end 早 0.1-0.8s (造成重叠)。
   - 句长：短句(1.5-3s), 中句(3-5s), 长句(5-8s)。
   - 每一句的 start 必须大于等于上一句的 start。
5：隐性情境原则 (Show, Don't Tell)
对话内容必须符合场景背景，但 **严禁在台词中直接复读或生硬拼凑环境标签的关键词**（如直接说出"兴奋"、"喜忧参半"、"生活事务"、"夜间行车"等）。
环境标签只是为了给你提供一种 Vibe（氛围、动机、语气），你必须通过角色的**具体反应、闲聊细节和语气词**来"暗示"这些环境，做到自然流露。
6、如果你认为给定的环境/场景组合存在逻辑冲突（例如：'车辆刚出发'与'到达目的地'），请以合理的逻辑将其串联，或者微调其中一个设定以保证对话的合理性。"
例如：抽到了"车辆刚出发"和"倒车或入库"。大模型可以修正为："车辆刚出发，但发现走错路了，需要倒车掉头"

【错误示范 - 机械复读标签】
- 场景标签：夜间行车_起雾_兴奋愉悦
- 错误台词："是啊，夜间巡航变道，起雾了，调整功能后感觉好兴奋。" (太假，像机器人在朗读设定)
```

### 4 下一步优化方向

**1 场景抽取策略改变**

目前环境池与任务池的抽取概率是均等的。但真实世界中，“稳定巡航”或”早高峰拥堵”的发生频次远高于”刚出发”或”突发复杂路况”。

下一步计划查阅相关交通统计文献或引入真实路测统计数据，为环境池中的各项标签赋予合理权重，采用加权随机抽样，使合成数据的场景分布更贴近真实物理世界。

**2 环境池体系扩充**

重新审视当前的环境标签体系，评估是否需要增加更高维度的宏观变量，进一步优化环境池的构成。

**3 打分Prompt优化：引入 CoT + 维度拆解**

```
Prompt
"你是一个车载对话分析专家。请你评估当前对话是否需要车载语音助手主动介入。
请严格按照以下 3 个维度分别打分（每项 0-10 分）：
```

- **召回率质变**：原始 Qwen 召回率仅为 3.64%（在 55 次该响应的场景中漏掉 53 次，呈现”假死”状态）。微调后，**召回率跃升至 50.91%**，模型初步具备了从嘈杂背景音中抓取用户隐晦需求的能力。
- **精确率与特异度保障**：微调后模型的精确率达到 71.79%、，且依然维持了 89.32% 的较高特异度（Specificity），有效抑制了”误触发”和”乱插嘴”现象。

### 3 问题发现：数据极化

在进一步的调优实验中，我们发现模型的打分输出呈现严重的**“两极分化”**。

- **极化数据实例：**

```
真实分数: 0.25 | 预测分数: 0.85
真实分数: 0.75 | 预测分数: 0.85
真实分数: 0.85 | 预测分数: 0.85
真实分数: 0.15 | 预测分数: 0.85

真实分数: 0.25 | 预测分数: 0.15
真实分数: 0.85 | 预测分数: 0.15
真实分数: 0.15 | 预测分数: 0.15

真实分数: 0.15 | 预测分数: 0.25
真实分数: 0.85 | 预测分数: 0.25
真实分数: 0.4 | 预测分数: 0.25
```

预测分数中只存在0.85 0.25 0.15三种类别。

- **原因分析：**

**1 教师模型偏好**：作为标注方，大模型自身在打分时极其自信，给出的标签数据本身就缺乏中间平滑过渡值（粒度是0.05，并且集中在0.15 0.25 0.4 0.75 0.85）

**2 生成式损失函数的局限**：当前的 SFT 采用交叉熵损失，模型将 `0.8` 和 `0.6` 视为两个完全无关的”独立词汇”。在分类概率的逼迫下，模型倾向于记住”非黑即白”的映射关系，导致输出退化为一个僵硬的二（三）分类开关，丧失了通过微调阈值来灵活控制系统灵敏度的能力。

### 4下一步优化方向

**1 转为回归式架构**

采用均方误差（MSE Loss）直接拟合连续数值。模型将真正理解”程度”的概念，使输出的迫切度得分呈现平滑的连续分布，从而允许业务端通过精细拨动阈值（如设定 0.55 或 0.65）来平衡召回率与误触发率。

**2 数据层面**

结合前述数据生成部分的规划（CoT + 维度拆解 Prompt），在预生成的 10000 条切片中，强制注入平滑的中间态分数，从数据源头喂给回归模型更高质量的连续值标签，以打破极化瓶颈。

## 被动响应复用问题

目前我们理解的被动响应，是用户明确唤醒后的多轮交互，例如：

user：理想理想，请为我打开空调

**model：好的，请问我需要为您把空调调到多少度**

**user：26度吧**

**model：好的，已为您打开空调，温度为26度**

而主动响应，是系统在未被唤醒的情况下，根据对话内容自发判断是否需要介入，例如：

user：我好热啊

model：我听到您说好热，需不需要我为您打开空调

user：好的，帮我打开

**model：好的，请问我需要为您把空调调到多少度**

**user：26度吧**

**model：好的，已为您打开空调，温度为26度**

如以上例子展示：

当主动响应启动后，一旦进入多轮交互（例如需要澄清温度等槽位信息），后续的对话模式与被动响应高度一致——本质上都是在进行”信息澄清-执行确认”的多轮对话。

### 两个方向

1主动响应是否只需要负责”首轮触发”（意图识别+响应时机确定+首句响应生成），然后将上下文打包交给成熟的被动响应模型，由它来完成后续的澄清与执行？ 这样设计的考虑是： 被动响应模块在澄清、多轮对话上已经比较成熟，是否可以复用成熟模块； 但主被动之间的上下文可能不完全一致（主动介入前有一段”监听”到的对话历史），但可以通过简单的实验来验证这种拼接是否可行。

2 如果主动响应需要负责全程 如果定义主动响应需要覆盖从触发到对话结束的全过程，那么我们的数据集构建就需要包含这种”主动触发+多轮澄清”的完整交互序列。这种情况下，可能需要借助现有的被动响应模型来辅助生成这类数据（例如让被动模型模拟后续的澄清轮次）。

# 2026/04/01进展

本周我们在若干约束条件下完成了**主动响应系统的搭建**，并基于新流程生成的 10,000 条数据进行了**模型微调**。具体约束如下：

1. 暂不处理用户打断；
2. 仅处理主动响应的首轮触发（意图识别 + 响应时机确定 + 首句响应生成），不将模型自身回复纳入上下文；
3. 仅考虑单会话（Session）内的主动响应，通过沉默阈值清空缓冲区，后续可能引入跨会话样本；
4. 对于用户抢话情况，当前处理方式与正常对话一致，未引入抢话信息。

## 主动响应系统

### 1 多轮对话数据预处理（`data_processor.py`）

- **上下文构建**：实现了对 Session 级别对话的遍历，能够精准提取当前轮次（Turn）的文本，并自动拼接历史对话记录以构建完整的上下文（Context）。
- **时间戳映射（Timestamp Mapping）**：建立了基于时间轴的索引系统。为每句话提取了 `end_time`（当前话音结束时间）和 `next_start`（下一句话音开始时间），为后续的”防打断”播报决策提供了精确的物理时间基准。

### 2 LLM 意图评估引擎（`llm_utils.py` & `pipeline.py`）

- **定制化 Prompt 设计**：构建了专用于车载场景的意图评估器。针对车内高频的复杂场景（如表达不适、隐性需求、车辆操作等），制定了 0.0 到 1.0 的细粒度评分规则。

### 3 智能播报决策模块（`pipeline.py`）

实现了一个 `broadcast_worker`，用于模拟真实的对话时间轴流转，并输出最终的交付状态（Delivery Status）：

- **状态 1：静默观察 (无需响应** `*no_need**`**)**。当 LLM 评估的紧急程度（Score）低于设定阈值（当前设为 `0.7`）时，系统判定当前对话为车内成员的纯闲聊或无明确意图的自言自语。此时系统选择保持克制与静默，绝不贸然打扰舱内氛围。
- **状态 2：主动避让 (响应被打断** `*interrupted**`**)**。当语义分数达标（系统认为应当提供帮助），但在系统规划的开口时机（当前话音结束 + `800ms` 黄金静默等待期）内，物理时间轴上检测到用户又开始了新的表述（即 `next_start < plan_time`）。此时系统展现出极高的交互素养，果断舍弃已生成的回复，主动取消播报，完美规避了人机”抢话”的尴尬场景。
- **状态 3：精准触达 (成功送达** `*delivered**`**)**。当且仅当同时满足”高度意图阈值”与”平稳度过 800ms 防打断物理校验”两大苛刻条件时，系统将在最完美的社交时机切入，最终成功输出高质量的 `response` 闭环服务。

* 解决方法：为段式样本重新打时间戳

### 时间戳规则重打

时间戳生成从给定规则到

将自然语义理解与切分交给大模型，让它读取生成的1000个session，对每一个间隔二分类，不是间隔停顿就是重叠，如果是间隔，再按照上述三种停顿做三分类，使用 Python 的 `random.uniform(min, max)`，根据大模型给的类别，在指定区间内生成浮点数（如 0.63s, 1.12s），作为间隔时长

**基准时长**：用规则打底——`句子时长 (duration) = 字符数 * 0.3秒`。

**时序兜底dd（重叠）**：在计算重叠时，加入 `max(min_allowed_start, calculated_start)`，强制规定下一句话最多只能倒退回上一句话刚开始 0.5 秒之后的位置，防止时序崩盘

**重新排序索引**：最后，抛弃最初的乱序，严格按照每一句的 `end时间戳`进行全局升序排序，并重排 `index`。

![重打时间戳之后的间隔与重叠分布](attachments/961ec404-6e6a-43cd-825f-7fb54580a2eb.png ” =2800x1600”)

重叠效果不好的原因：

大模型对绝对时间的感知极差。当你让它输出重叠秒数时，它根本不知道上一句话具体有多长（比如上一句只有 0.9秒，它敢输出重叠 1.5秒）。并且，它依然只会输出 `0.5`、`1.0` 这种极其生硬的整数/半整数，导致即使有重叠，分布也是离散的针状，而不是平滑的曲线。

改进方案：

把重叠变为”比例”，并将随机权彻底收回给 Python脚本，优化的核心思路是：**大模型只准说”这里有没有重叠”，不准说”重叠多少秒”。具体的重叠时间，由 Python 根据上一句话的总长度，按随机比例动态切入，**如果大模型说是 Overlap，我们就让当前这句话，随机切入上一句话结尾的 20% 到 60% 处。

## **异步**模拟和处理车载流式语音交互

引入了一个贴近真实自动驾驶/语音对话系统的组件：`SpeechActivityTimeline`（语音活动时间轴）。

相比于之前简单地用 `end_time` 和 `next_start` 做数学大小比较，这个新版本的决策机制变成了**真正的流式时序模拟（Streaming Simulation）**。

### 1. 任务注入与并发推理 (Producer & Consumer)

前半段逻辑与之前类似：

- 主线程通过 `add_unit()` 把对话轮次（Unit）塞入 `input_queue`。
- 由 `num_consumers` 指定数量的工作线程并发地从队列拿任务，调用 `call_deepseek` 进行推理。
- 拿到结果后，组装成 `ResponseUnit`（包含评分 `score`、回复内容、结束时间 `end_time` 等），放入 `model_output_queue`。

### 2. 真实的时序模拟决策 (Broadcast Worker)

改造 `_is_interrupted_during_silence_window` 方法，它不再是”一眼看穿未来”（拿到下一句话什么时候开始），而是**像一个真实的系统一样，一帧一帧地”监听”当前环境。**

- **第一步：过滤低意愿响应。** 从模型输出队列拿出结果后，先看 `score` 是否达到了 `URGENCY_THRESHOLD`（紧急度/播报意愿阈值）。没达到直接判为 `no_need`。
- **第二步：启动”帧级别”的静音侦测 (Tick-based Silence Checking)。** 如果分数达标，系统进入静音等待期（比如设定 `SILENCE_GAP_SEC` 为 800毫秒）。模块会按照设定的采样率（`tick_seconds`，比如每 0.1 秒一帧）将这段时间切成多个小检查点（steps）。
- **第三步：基于 Timeline 查表与 Sleep 模拟。** 在每一个 check 节点：
    1. 程序会通过 `time.sleep(sleep_seconds)` 真正地等待一段时间（受 `playback_speed` 控制，比如可以 10 倍速快进模拟）。
    2. 醒来后，向外挂的 `speech_activity_timeline` 模块查询：在当前这个绝对时间戳 `check_timestamp` 上，车厢里到底有没有人说话？（`get_value` 返回 `True` 或 `False`）。
- **第四步：决策落槌。**
    - 如果在等待期内的**任何一帧**，发现有人说话（`get_value` == `True`），说明在这800毫秒内用户出声了。系统立刻判定：计划取消，状态标记为 `*interrupted**`（被打断）。
    - 如果熬过了所有的 step，环境依然保持安静，说明系统成功找到了合适的空窗期，状态标记为 `*delivered**`（成功播报）。

## **并行输入调研**

当前我们的策略是追踪所有用户的最近的一个主动响应需求

从多用户任务型对话研究来看，**Yohan Jo( 2023) 等 *Multi-User MultiWOZ: Task-Oriented Dialogues among Multiple Users*** 提出 **multi-user contextual query rewriting**，先把两个用户之间的协商聊天改写成可供单个任务型对话系统直接消费的任务查询，再沿用原有对话框架继续处理。论文的实验也表明，这种 rewriting 可以帮助下游的 dialogue state tracking。这个结果说明，在较新的多用户任务型对话论文里，落点是对多用户输入线索进行保留、筛选和重组

多方对话的方向上看，**Inoue 等人在 2025 年的论文 *An LLM Benchmark for Addressee Recognition in Multi-modal Multi-party Dialogue*** 把 addressee recognition，即”当前这句话是对谁说的、下一轮该由谁接话”，当作多方对话里的关键任务单独拿出来评测。与此同时，**2024 年的论文 *Do LLMs suffer from Multi-Party Hangover? A Diagnostic Approach to Addressee Recognition and Response Selection in Conversations*** 也把多方对话中的 addressee recognition 和 response selection 作为核心诊断对象。它们都说明，即使到了近两年的 LLM 语境下，多人交互里的核心问题是**如何正确识别当前输入应关联到哪一个对话对象、以及系统应如何选择后续响应**

假设车内有两位用户：

> 用户 A 先说”帮我导航到最近的充电站”
> 
> 
> 系统尚未完成响应
> 
> 随后用户 B 插入一句”空调调到 22 度”
> 
> 紧接着用户 A 又补充”要支持快充的”。
> 

在这种情况下，如果把”并行输入”理解成真正意义上的并行多回复，那么系统需要同时管理多条输出时机，复杂度较高。

如果参考 Jo 等人的 multi-user query rewriting 思路，以及 Lin 等人关于多人协作决策中约束持续整合的视角，那么一个更自然的做法是：**输入流仍按时间顺序接收，但系统内部按用户维护未完成需求**。这样，A 的需求链条会被保留为”导航到充电站”并继续被”支持快充的”这一补充约束更新；B 的需求则作为可较快闭环的控制指令单独维护。

这样处理后，系统外部看起来仍然是顺序响应，但内部实际上已经保存了多用户、多条未完成需求并存的状态。这个推导也说明，在我们已经拿到说话人和时间戳之后，真正关键的问题不再是”谁说了什么”，而是”**哪些用户需求在当前时刻仍然有效，并且仍需系统继续处理**”。

因此，当前一个的调研结论是：

> **现阶段可以先把并行输入近似理解为：顺序式输入处理 + 按用户维护(最近一次)尚未完成的主动响应需求。**
>

## 段式数据理论补充

### 关于三种停顿的调研

[campione-veronis.pdf 108987](attachments/10ef9f38-3388-44a9-95c0-8ac55e38eb41.pdf)

这篇论文《A Large-Scale Multilingual Study of Silent Pause Duration》（Estelle Campione & Jean Veronis）主要研究了五种语言（英语、法语、德语、意大利语、西班牙语）中静默停顿的时长分布，基于约6000个停顿、总计5.5小时的朗读和自发语音数据。

1. 停顿时长呈三峰分布，而非正态分布，因此常用的统计方法（如ANOVA）不适宜直接使用。
2. 三种停顿类型：
    
    （1）短停顿 (Brief pauses, <200ms)
    
    论文指出，在之前的研究中存在一种隐性假设，即这种极短停顿的作用主要是生理和呼吸上的，可能与发音生理活动（如呼吸、辅音闭塞段）相关。但作者紧接着强调，目前并没有大规模的研究能证明它们完全没有”结构或句法作用，它们并非纯粹生理性的，也可能有结构或句法作用。在数据上，这种短停顿在朗读和自然口语中都存在，峰值大约在 100-150 毫秒左右 。
    
    （2）中等停顿 (Medium pauses, 200-1000ms)
    
    这种停顿是数据中占比最大的部分，在朗读语音和自然口语中都会大量出现，与句法边界、语流中的自然分隔相关 。它的分布峰值集中在 500-600 毫秒左右 。作者在文中没有直接用”句法”或”换气”来命名它，而是将其作为人类维持正常语流时最普遍的常规停顿状态。
    
    （3）长停顿 (Long pauses, >1000ms)
    
    作者明确指出，大于1000毫秒的长停顿仅仅出现在自然口语中。
    

## ![Distribution of pause durations (ms, spontaneous speech)](attachments/f682423e-d8a1-4801-9dbc-9566a2fb9629.png ” =534x357”)

比朗读语音更偏斜，长尾更明显（出现>1000ms 的停顿），突出自发言语中长停顿显著增多，分布形态与朗读语音差异明显

## 主动响应系统

### 旧pipeline存在的问题及相应优化

1. 意图推翻与自我否定

这是自然口语中最频发的情况，用户话说到一半突然改变了主意。此时response不仅不能合并，还需要被彻底”反转”或丢弃。

挂起的response：“好的，已为您导航至汉堡王。”

上文

> A：我有点饿了，我们去汉堡王吃午饭吧
> 

下文：

> B：“……呃，还是去麦当劳吧。”
> 

模型动作 ：discard（）

最终Response：“好的，已经为您导航至麦当劳。”

2. 澄清

用户先抛出一个粗略的任务框架，在系统响应前又补充了具体的约束条件。此时需要将约束条件融合进原任务。

挂起的response：“好的，已为您打开主驾车窗。”

上文：

> A:有点热，不然开点窗户把
> 

下文：

> B:稍微开一条缝就行，别全降下来，风太大了。”
> 

大模型 (Prompt2) 的思考逻辑：指令的实体（主驾车窗）一致，但动作的颗粒度（全开 -> 开一条缝/10%）发生了变化。后文是对前文的细化。

动作：Merge

最终Response：“好的，已经帮您把车窗留了一条缝隙。”

3.任务自闭环与物理抢占

用户在等待期间，自己通过物理按键完成了操作，或者意识到不需要系统介入了。此时系统必须做到”高情商退场”。

挂起的response：“马上为您调高空调温度。”

上文：

> A：今天天气怎么这么热，开会空调吧。
> 

下文：

> A：“算了你别弄了，我刚好摸到这个旋钮了，我自己拧一下。”
> 

大模型 (Prompt 2) 的思考逻辑：用户明确表示任务已通过其他物理途径完成，系统若再抢话确认会显得极其多余且啰嗦。

动作 ：discard

最终 Response：（大模型输出极低的 Score，Response 为空，系统直接清空该草稿，保持无感静默。）

4. 并行指令堆叠

用户一口气下达了多个毫无关联的指令，因为中间有思考停顿导致第一个指令生成了草稿，随后别的用户又补上了第二个指令。

挂起的 Draft：“天窗已为您关闭。”

> A:今天太阳怎么那么晒，把天窗关了吧。
> 

下文 ：

> B：“好无聊，放一首适合睡觉的纯音乐吧。”
> 

大模型 (Prompt 2) 的思考逻辑：两个意图互不冲突（控制车辆硬件 vs 多媒体播放），属于并行任务。需要用自然的人类口语将它们串联起来。

动作：Merge

最终Response：“天窗已经关好了，马上为您播放助眠的纯音乐，祝您休息得好。”

所以增加一个需求通道

为了不丢掉响应，优化方向：

- 引入多线程：为每个乘客单独维护一个通道

- **ASR 文本流摄入**：底层的语音识别（ASR）模块将车内的音频实时转化为文本，并打上时间戳（Start/End）。
- **公共拓展缓存区 (Global Text Buffer)**：作为全局的”真值表”，无差别地接收并按时间线存储所有转录文本。它确保了无论车内发生了多么复杂的抢话或重叠，底层的数据时序永远是完整的。
- **动态通道隔离**：系统从公共缓存区提取数据，并根据发音人（Speaker A, B 等）进行切片包装，送入相互独立的判定通道，捕捉到新乘客介入时，系统会自动开辟”通道 N”，适应座舱内乘员动态变化的环境。

![](attachments/b113ceeb-f47b-4fcb-98e1-6c58d8ac89a9.png ” =1086x932”)

**核心功能**：决定系统在收到用户指令后，是立刻发声、静默丢弃，还是”欲言又止”并开启下文监听。

- **Qwen 初次打分 (Prompt 1)**：单人通道的切片数据送入模型，模型仅凭当前上下文输出当句的介入急迫性（Score）和回复内容（Response）。
- **门槛校验与帧轮询检测**：
    - 首先拦截低分（Score < 0.7）的无效回复或闲聊，直接静默丢弃。
    - 对于高分回复，系统不立刻发声，而是进入**800ms 静默等待期**。在这 800ms 内，系统以高的频率（0.1s一帧）轮询检测 `get_value`。
- **物理分流点**：
    - **Delivered** ：如果平稳熬过 800ms，说明用户话已说完，系统立刻调用 TTS 播出响应。
    - **Interrupted** ：如果在轮询中检测到用户再次出声（抢话），系统果断取消播报计划。随后，这个未能播报的 Response 会作为”遗留问题”被挂载到需求列表中，并同步开启下文追踪列表，开始记录用户抢话后的新内容。
    - ![](attachments/53fb4b15-6e43-4606-87f8-f2376aee7760.png ” =1786x899”)

**核心功能**：作为系统的”垃圾回收与语义消解”机制，处理那些被打断的草稿，确保最终回复智能且得体。

- **物理时钟触发**：当车内再次陷入绝对安静，沉默域达到 800ms 时，系统自动唤醒。
- **上下文全量装载**：系统像”三明治”一样，将之前保留的**历史旧上文**、挂起的 **response**，以及最新收集到的**下文追踪语境**严密拼接到一起，送入大模型。
- **大模型语义消解 (Prompt 2)**：模型基于完整的上帝视角，评估中间那个未能播报的草稿是否还有价值：
    - **动作：Discard (丢弃)**：如果用户在下文中明确推翻了先前的需求，模型判定草稿失效，彻底清除挂起数据，并根据新需求重新响应。
    - **动作：Merge** ：如果用户的下文是对原需求的补充或细化（例如：“空调风量再调小点”），模型将旧意图与新需求完美融合，输出一句极其自然且包含所有条件的最终 Response，送入 TTS 播报。
    - **动作 ：dlivered：**模型认为还要说，且这就是当前唯一的未决需求，直通 TTS 扬声器。

在三种状态的分类中完全依赖 Prompt 的 Few-shot来做复杂逻辑推演不够稳定，后续计划生成1000组数据：

- 意图推翻与自我否定 (Discard)：约 300 组。这类场景最多，句式也最复杂（比如”呃”、“算了”、“不对”等人类纠结口癖），需要大量数据让模型学会识别”否定转折词”。
- 澄清与约束细化 (Merge)：约 350 组。这是难度最高的一类，因为模型需要提取 A 的实体和 B 的约束并重新生成，极度考验模型的自然语言生成能力。
- 并行指令堆叠 (Merge)：约 200 组。重点在于跨领域任务的拼接（如车辆控制+多媒体），数据构造时要刻意制造毫无关联的指令跨度。
- 任务自闭环与物理抢占 (Discard)：约 150 组。特征相对明显，主要是识别用户话语中的物理动作描述（“按了”、“拧了”、“关了”），教会模型适时”闭嘴”，给出极低的 Score。

对不同任务不一样：几百毫秒 汽车知识问答几百毫秒 大模型相关的几百毫秒之内 简单的闲聊简短的response比较短 写报告之类的耗时比较长 首token几百毫秒之内
